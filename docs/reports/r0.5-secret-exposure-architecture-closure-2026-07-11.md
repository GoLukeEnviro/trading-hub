## Rotation Process (for future incidents)

Documented for reuse — same transactional sequence used in E0:

1. Redact any public-facing exposure immediately (GitHub issue, comment, or PR body).
2. Inventory all consumers of the affected credential (containers, env files, compose references) — names and paths only, never log values.
3. Generate the new credential and store it with restrictive permissions, owner-only.
4. Update each consumer individually.
5. Reload or recreate affected services in a controlled order.
6. Run functional tests to confirm the new credential works.
7. Revoke the old credential at the provider, where applicable.
8. Search for residual occurrences in repo history, reports, and logs — do not reproduce values in the search output.
9. Write an incident report without reproducing secret values.

Rule: a compromised credential is never reactivated. If a rotation attempt causes problems, issue a second new credential — never revert to the compromised one.

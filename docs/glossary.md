# Trading Hub — Glossary

> Canonical glossary for Trading Hub terminology. Agents and operators
> should use these definitions for consistent communication.

---

## SI-v2 Self-Improvement System

### AUTONOMOUS_DRY_RUN
The current SI-v2 operating mode (since ADR-2026-07-01). Policy-gated,
canary-first, allowlist-based dry-run apply. No human gate per iteration.
Human approval required only for live-mode transitions and emergency
overrides. See `docs/decisions/ADR-2026-07-01-si-v2-autonomous-dry-run-loop-live-target.md`.

### ShadowProposal
A proposed change generated by the SI-v2 loop from measurement analysis.
Proposals are scored, validated, and may be applied to the canary bot
(policy-gated). Each proposal carries evidence bundles, gate verdicts,
and a deterministic ID. Proposals are audit-logged via ShadowLogger.

### RuntimeEffectProof
Post-apply verification that a change was correctly applied and the
target bot is operating as expected. Proof status can be GREEN (verified)
or RED (mismatch detected). Generated by `proof.py` in the SI-v2 apply chain.

### Measurement Decision
Post-apply evaluation by the decision engine (`decision_engine.py`) that
determines whether to KEEP (continue), EXTEND (observe longer), or
ROLLBACK the applied change. Based on quantitative metrics over a defined
measurement window.

### KEEP_CANARY_OVERLAY
A measurement decision outcome meaning the applied change is performing
within acceptable bounds and should remain active on the canary bot.
Does NOT imply live deployment.

### ROLLBACK_RECOMMENDED
A measurement decision outcome (or C4 gate decision) indicating that the
applied change has breached thresholds (e.g., max_drawdown_pct) and
should be reverted. Triggers rollback execution and blocks D1 progression.

## Safety Components

### RiskGuard
The preferred risk authority for the trading system. Defines safety gates,
accept/watch-only/block-entry verdicts, and production risk limits. Currently
a specification (`docs/specs/production-risk-limits-spec.md`) with partial
implementation in shared fleet code. See `docs/specs/runtime-safety-contract.md`.

### ShadowLogger
The append-only JSONL audit trail required for all decision/write
operations in the SI-v2 loop. Records every proposal, gate verdict, apply
action, and measurement decision. Lives in `orchestrator/logs/`.

### Kill Switch
The central safety choke point (`freqtrade/shared/kill_switch.py`). Three
modes: NORMAL (operation allowed), HALT_NEW (block new entries, keep
positions), EMERGENCY (block entries, signal position close). Atomic
file-based state with auto-clear timer and drawdown guard integration.

### HALT_NEW
A kill switch mode that blocks new trade entries but holds existing
positions. Used for elevated risk conditions. Triggered at 12% drawdown
(default).

### EMERGENCY
A kill switch mode that blocks new entries and signals position close.
Used for drawdown breach or manual emergency. Triggered at 18% drawdown
(default).

## Fleet and Deployment

### Canary
The canary bot (`freqtrade-freqforge-canary`), an independent FreqForge
instance used as the first target for SI-v2 dry-run apply changes. All
mutations are applied to the canary first, measured, and only promoted
to the fleet after successful validation.

### Control Bot
A reference bot (typically `freqtrade-freqforge`) used as a baseline
for comparison when measuring the effect of canary apply changes.
The control bot does not receive any SI-v2 mutations.

### Freqtrade Fleet
The set of dry-run Freqtrade bot containers: FreqForge, Regime-Hybrid,
FreqForge-Canary, and FreqAI-Rebel. All run `dry_run=true`. Orchestrated
via Docker Compose.

## Operating Levels and Governance

### L2
Operation level: Safe dry-run write. Reports, docs, manifests, non-destructive
artifacts. Usually does not need explicit approval.

### L3
Operation level: High-risk. Config mutation, Docker restart/rebuild,
data deletion, strategy/risk changes, credentials, live trading. Requires
explicit human approval.

### Approval Marker
A tracked token (e.g., `APPROVED_LIVE_CANARY_TRANSITION`) in
`docs/decisions/` or an issue comment that authorizes a specific action.
Markers are scope-specific, time-limited (7 days), and never generic.
See `docs/decisions/README.md` for the full index.

### Live Target Architecture
The intended end-state where the system can execute live trades under
controlled conditions. NOT currently enabled. Requires D1 approval,
`APPROVED_LIVE_FLEET_ROLLOUT` marker, and all live-readiness gates
passing. See GitHub Issue #423.

### LIVE_FORBIDDEN
The default state machine position. No real orders, no live keys, no
`dry_run=false`. The system must be explicitly transitioned out of this
state via documented approval gates.

### LIVE_APPROVED
State indicating that explicit human approval exists with documented
scope, limits, exchange, bot, strategy, timeframe, and rollback plan.
Required before any `dry_run=false` can be set.

### D1
Track D, Step 1 — Live Fleet Rollout. The final gate for deploying
live trading across the fleet. Currently **BLOCKED** by C4
`ROLLBACK_RECOMMENDED` and missing `APPROVED_LIVE_FLEET_ROLLOUT` marker.
See GitHub Issue #423.

## Deprecated / Historical Terms

### HUMAN_GATED_CANARY_APPLY_PHASE_1
Historical SI-v2 controller mode (Phase 1). Human-gated apply was available
for the canary bot via explicit approval per iteration. **Superseded for
dry-run by ADR-2026-07-01** (AUTONOMOUS_DRY_RUN). Preserved as historical
context.

### PAUSED / L3_REPOSITORY_ONLY
Historical SI-v2 controller state. The controller could only perform
repository-level operations, no runtime apply. **Superseded by
AUTONOMOUS_DRY_RUN** (ADR-2026-07-01).

### Primo / Bridge
Historical signal pipeline components. PrimoAgent and Hermes-Primo Bridge
handled signal filtering and routing. **Decommissioned in Phase 44-45.**
Replaced by the SI-v2 autonomous dry-run loop. `primo_signal.py` remains
in `freqtrade/shared/` as a legacy signal filter and kill-switch
integration boundary.

# ADR-2026-06-27: Controlled Self-Improvement — Human-Gated Canary Apply (Phase 1)

## Context

SI-v2 has operated in read-only / evidence-only mode since its inception. ShadowProposals
are generated by the Active Cycle, but the system has had no controlled path to apply
even safe parameter changes to a running bot.

Issue #363 and PR #371 introduce a **Phase 1 Controlled Apply Actuator** that bridges
this gap without compromising safety, auditability, or the dry-run-only trading posture.

## Decision

Introduce a **human-gated, canary-first, token-gated Controlled Apply Phase 1** with
the following architecture:

1. **`check_readiness()`** — read-only runner that evaluates all 8 gates (canary, safe
   parameters, kill-switch, RiskGuard, human approval, token, cooldown, dry_run)
   and returns a `ReadinessReport`. No side effects.
2. **`execute_apply()`** — calls `check_readiness()` first; only writes the safe
   parameter overlay if all gates pass. Returns `SHADOW_OVERLAY_WRITTEN` status.
3. **`run_controlled_apply_canary()`** — public entry point that wraps `execute_apply()`.

The actuator integrates with the existing `apply_actuator` stack (`controlled_apply.py`,
`policy.py`, `proof.py`) via `ApplyActuatorResult` delegation.

## Scope

- **Target bot:** `freqtrade-freqforge-canary` only
- **Parameter scope:** 6 safe parameters defined in `SAFE_PARAMETERS` (`rsi_period`,
  `stoploss_pct`, `take_profit_pct`, `stake_factor`, `max_open_trades`, `cooldown_candles`)
- **Overlay write path:** orchestrator-side storage (`/opt/data/profiles/orchestrator/config_overlays/canary/`)
- **Status:** `SHADOW_OVERLAY_WRITTEN` — does not claim runtime visibility

## Non-goals

- No autonomous apply
- No runtime apply (Docker-visible overlay requires separate L3 approval)
- No strategy code changes
- No non-canary apply
- No Docker restart, Compose mutation, or cron mutation
- No `dry_run=false`
- No live trading

## Safety invariants

- Mutations require human gate: `requires_human_approval=True` enforced in code
- Mutations require L3 token: the activation environment variable must match the approved value
- Kill-switch `HALT_NEW` or `EMERGENCY` blocks all apply actions
- RiskGuard must report `PASS` (conservative: `None` blocks)
- Cooldown: max 1 apply per 7 days. Corrupt cooldown state blocks (fail-closed)
- `dry_run` gate is unconditional: missing config blocks
- Parameter values are range-validated via `validate_safe_parameter()` with
  `_PARAMETER_RANGES` bounds
- ShadowLogger writes are required before overlay write; log failure blocks
- `mutation_counter_should_increment: False` — always `False` in Phase 1 (no runtime proof)
- `measurement_allowed: False` — always `False` in Phase 1
- `runtime_visible: False` — overlay written to orchestrator-side storage
- `runtime_proof_status: "NOT_RUN"` — no Docker-level runtime proof executed

## Human approval model

1. Operator identifies a candidate (e.g. `f68a031923d0`, cooldown_candles 3→4)
2. Operator runs `check_readiness()` — produces read-only `ReadinessReport`
3. If `ReadinessReport.ready == True`, operator may proceed to `execute_apply()`
4. `execute_apply()` requires:
   - The activation environment variable must be set to the approved value
   - Pre-apply config snapshot
   - `riskguard_status="PASS"` (from external RiskGuard system)
5. After overlay write, the operator must manually verify the bot can load the change
   (separate L3 step, not automated in Phase 1)

## Canary-first rule

Only `freqtrade-freqforge-canary` is accepted as a target in Phase 1. All other bot IDs
are unconditionally blocked at the first gate. This restriction may be relaxed in a
future phase after the canary path is proven in shadow/readiness mode.

## Rollback model

- Before any overlay write, a rollback plan is created containing the pre-apply config
  snapshot, the overlay path, and restore instructions
- Rollback is file-level in Phase 1: restore the base config and remove the overlay file
- Bot restart (if needed) is a separate L3 action, not automated by the actuator
- Rollback plan is persisted alongside the overlay and cooldown state

## Measurement / mutation counter rules

- Mutation counter must NOT increment in Phase 1 (no `RuntimeEffectProof` GREEN)
- Measurement must NOT start in Phase 1 (no `APPLIED_WITH_RUNTIME_PROOF`)
- These invariants are enforced by hardcoded `False` defaults in `ControlledApplyDecision`
- A future Phase 2 may activate measurement after:
  - Runtime proof GREEN (via `proof.py` / `policy.py` composite proof)
  - Token-gated controlled apply with `APPROVED_WITH_RUNTIME_PROOF` status
  - Explicit operator approval to enter measurement mode

## Risks

| Risk | Mitigation |
|------|------------|
| Operator error in apply scope | 8-gate pipeline blocks unless all conditions are met |
| Corrupt cooldown state | Fail-closed: `_corrupt=True` → cooldown active → blocks |
| Missing RiskGuard integration | Conservative gate: `None` blocks. External integration required. |
| Overlay not runtime-visible | `runtime_visible=False` explicitly documents this. Not automated. |
| ShadowLogger failure | `log_shadow_events()` has fallback JSONL writer; write failure blocks overlay write |
| Parameter key mismatch | `SAFE_PARAMETERS` maps to SI-v2 candidate keys. Real Freqtrade key mapping needed for `stoploss_pct` etc. |

## Validation

- 40 dedicated tests for the Phase 1 actuator
- 158 combined apply-actuator tests passing
- Ruff clean, compileall clean
- PR #371 squash-merged to `main` at `45be0d418804caff92439beb859e68a43d1f7a20`

## Rollback of this decision

To revert Phase 1:
1. Do not use `execute_apply()` — it requires explicit operator action
2. The actuator can be removed by reverting PR #371
3. No runtime state is affected by the actuator code itself
4. No bots, containers, configs, or cron jobs are mutated by the source code
